## 1. 目的（何を作るのか）

PDF化した約500冊の蔵書を、**自分専用の調べ物に使える検索・質問システム（RAG）**にする。

- 用途: **個人の調べ物・リサーチ専用**（公開・他者提供はしない＝著作権上も私的利用の範囲）
- やりたいこと: 「全蔵書を横断して質問 → 関連箇所を**出典付き**で答えてもらう」
- やらないこと: ファインチューニング（知識注入には不向き・出典が出せない）。本プロジェクトはRAGで構築する。

### なぜRAGか（方針の根拠）
- ファインチューニングは事実知識の注入に弱く、出典も出せず、本が増えるたび再学習が必要。
- RAGなら **出典提示・追加が容易・ハルシネーション減** のメリットがあり、個人の調べ物に最適。

## 2. 全体像（パイプライン）

```
PDF原本（SSD）
   │  ① OCR（縦書き和書対応のエンジンで取り直す）
   ▼
テキスト + メタデータ（書名 / ページ番号）
   │  ② チャンク分割
   ▼
ベクトル化 → ベクトルDB（Chroma）
   │  ③ インデックス構築
   ▼
質問 → 関連箇所を検索 → ④ Claudeが出典付きで回答
```

スクリプトは `scripts/` に、各段を分けて置いてある:

| 段 | スクリプト | 役割 |
|----|-----------|------|
| ① | `scripts/01_ocr.py`   | PDF → ページ画像 → OCRテキスト（書名・ページ付き） |
| ②③ | `scripts/02_index.py` | テキスト → チャンク分割 → ベクトル化 → DB登録 |
| ④ | `scripts/03_query.py` | 質問 → 検索 → 出典付き回答 |

詳細な技術設計は **[docs/ARCHITECTURE.md](docs/ARCHITECTURE.md)** を参照。

## 3. 重要な前提・判断（確定事項）

| 項目 | 決定 | 理由 |
|------|------|------|
| 構築方式 | **RAG**（ファインチューニングしない） | 知識活用・出典・更新容易性 |
| 用途 | **個人の調べ物専用** | 著作権の私的利用範囲に収める |
| OCR | **縦書き対応エンジンで取り直す** | 既存OCRは縦書きで品質が悪いため。原稿画質は良好と確認済み（再スキャン不要） |
| OCRの割り切り | まず安いOCRで全冊、深く読む本だけ高精度OCR | 500冊全部を完璧にするのは費用対効果が悪い |
| 実データの所在 | **外付けSSD / ローカルPC**（このクラウドからは触れない） | セッション分離。重い処理はローカル側で実行する |

### 品質確認の結果（2026-06-09）
- サンプル（縦書き和書『BRAIN DRIVEN』の数ページ）をLLM-OCRで起こしたところ、**ほぼ100%の精度**。
- 原稿はデジタル品質でクリーン。**問題は原稿ではなく「縦書きに弱いOCRエンジン」だった**。
- → 結論: **再スキャン不要。縦書き対応エンジンに変えるだけで解決する。**

## 4. リポジトリ構成

```
PROJECT.md                  ← このファイル（司令塔）
docs/
  ARCHITECTURE.md           ← 技術設計（パイプライン詳細・OCR/埋め込みの選定）
  WORKFLOW.md               ← セッション間の運用ルール / 日報・ナレッジの仕組み
config/
  config.example.yaml       ← 設定テンプレート（コピーして config.yaml を作る）
scripts/
  lib/common.py             ← 共通処理（設定読込・パス・カタログ）
  01_ocr.py / 02_index.py / 03_query.py
  requirements.txt
knowledge/                  ← 蓄積する「知識資産」（←ここが本プロジェクトの価値の中心）
  catalog.csv               ← 蔵書カタログ（書誌メタデータ・処理状態）
  books/                    ← 本ごとの要約・読書ノート
  themes/                   ← 複数の本を横断したテーマ別ノート
reports/
  PROGRESS.md               ← 全体進捗（ライブ状態）
  daily/                    ← 日報（その日やったこと・気づき・詰まり）
```

> ⚠️ **実データ（PDF・OCRテキスト・ベクトルDB）はGitに載せない**（重い & 著作物）。
> `.gitignore` で除外済み。Gitで共有するのは**スクリプト・設計・ナレッジ（テキスト）だけ**。

## 5. セッション運用（要点）

- セッション同士はリアルタイム連携できない。**このリポジトリを“ハブ”にして連携**する。
- **重い実作業（500冊のOCR・DB構築）はSSDを触れるローカルセッションが担当。**
- このリポジトリ用セッションは**設計・スクリプト作成の工房**として使う。
- 連携の作法・日報・ナレッジの書き方は **[docs/WORKFLOW.md](docs/WORKFLOW.md)** に集約。
