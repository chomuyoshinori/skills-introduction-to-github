# 運用ルール — セッション連携 / 日報 / ナレッジの仕組み

このプロジェクトの肝は「**個々の本の知識を最大限に活用し、複数セッション間で共有する仕組み**」です。
そのための運用をここに定義します。全体像は [../PROJECT.md](../PROJECT.md) を参照。

---

## 1. セッション間の連携モデル

セッション（このリポジトリ用 / ローカルPC / SSD作業）は**リアルタイムには連携できません**。
代わりに、**このGitリポジトリを共有ハブ**にして非同期で連携します。

```
        ┌─────────────────────────────┐
        │   共有ハブ（このリポジトリ）      │
        │  PROJECT.md / docs / scripts  │
        │  knowledge/ / reports/        │
        └──────▲───────────────▲───────┘
               │ pull/push      │ pull/push
     ┌─────────┴────┐    ┌──────┴──────────┐
     │ ローカルSession │    │ このSession      │
     │ SSD実データを処理 │    │ 設計・スクリプト作成 │
     └──────────────┘    └─────────────────┘
```

### 役割分担
- **ローカルセッション（SSDを触れる）**: OCR・インデックス構築など**重い実作業**。
- **このリポジトリ用セッション**: 設計・スクリプト改良・ナレッジ整理。

### 鉄則
1. 作業を始める前に `git pull` して最新の `PROJECT.md` / `reports/PROGRESS.md` を読む。
2. 区切りがついたら **日報を書き**、`PROGRESS.md` を更新して `git push`。
3. 大きな方針変更は必ず `PROJECT.md` に反映する（口頭・記憶に頼らない）。

> 専任の「まとめ役セッション」は不要。**まとめ役はファイル（PROJECT.md / PROGRESS.md）が担う。**

## 2. 知識を最大限に活用する仕組み（4層）

単に検索できるだけでなく、**読んで得た知識が蓄積・再利用される**ようにします。
`knowledge/` 以下に4層で知識を貯めます。

| 層 | 置き場所 | 中身 | 誰が書く |
|----|---------|------|---------|
| ① 蔵書カタログ | `knowledge/catalog.csv` | 全本の書誌＋処理状態（OCR済/索引済） | スクリプト＋人 |
| ② 本ごとのノート | `knowledge/books/<slug>.md` | 1冊の要約・要点・引用・自分の所感 | 人（Claude支援） |
| ③ テーマ別ノート | `knowledge/themes/<topic>.md` | 複数の本を横断した論点・比較・統合 | 人（Claude支援） |
| ④ 日報 | `reports/daily/YYYY-MM-DD.md` | その日の作業・発見・詰まり | 各セッション |

### なぜこの4層か
- **カタログ**＝何を持っているかの一覧（重複・抜けの把握）。
- **本ノート**＝1冊から抽出した知識の結晶。検索の補助＆再読の起点になる。
- **テーマノート**＝本を跨いだ「気づき」を貯める場所。ここが知の付加価値の中心。
- **日報**＝時系列のログ。セッションを跨いで「どこまで・何を考えたか」を引き継ぐ。

これらは**テキストなのでGitで共有でき**、どのセッションからも読める＝知識が個人とAIの間で循環します。

## 3. 日報（デイリーレポート）の運用

**1日の作業の区切り、またはセッションの終わりに必ず1本**書きます。

- 置き場所: `reports/daily/YYYY-MM-DD.md`
- 雛形: `reports/daily/_TEMPLATE.md` をコピーして使う
- 同じ日に複数セッションが書くときは、同じファイルに追記（見出しでセッションを区別）

### 日報に書くこと（テンプレより）
1. **やったこと** — 処理した本・書いたコード・更新したノート
2. **発見・学び** — OCRのコツ、良かった本、知識の気づき（←ここを丁寧に）
3. **詰まり・課題** — エラー、判断に迷った点（次のセッションへの申し送り）
4. **次にやること** — TODO

> 日報は「作業ログ」であると同時に「**未来の自分／別セッションへの引き継ぎ書**」です。
> 特に **3. 詰まり** と **4. 次にやること** は、次に開いたセッションが迷わないための生命線。

## 4. 本ノート・テーマノートの育て方

### 本ノート（`knowledge/books/<slug>.md`）
- インデックスに入れた本は、`knowledge/books/_TEMPLATE.md` をコピーして1冊1ファイル作る。
- 要約は `03_query.py` や Claude に「この本の要点を出して」と頼んで下書き → 自分で吟味して残す。
- 後から検索でヒットした本を深掘りしたとき、所感をここに追記していく（育てる）。

### テーマノート（`knowledge/themes/<topic>.md`）
- 「モチベーション」「ストレス」など、横断テーマごとに1ファイル。
- 複数の本の主張を並べて**比較・矛盾・統合**を書く。RAGで複数本を引きながら作るのが効率的。
- ここが「500冊を串刺しにして初めて見える知見」の置き場所。

## 5. ファイル更新のチェックリスト（毎セッション）

作業の終わりに、以下を確認して `git push`:

- [ ] 日報を書いた（`reports/daily/YYYY-MM-DD.md`）
- [ ] `reports/PROGRESS.md` のチェックボックス／数値を更新した
- [ ] 方針が変わったなら `PROJECT.md` を更新した
- [ ] 新しく読み込んだ本があれば `knowledge/catalog.csv` と本ノートを更新した
- [ ] `git add` → `git commit` → `git push`
