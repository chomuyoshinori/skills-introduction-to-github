# AIエージェントの役割分担と「組織化」の判断

## 結論: いきなり「組織」を作らない

一点物の作り込みでは、**多数のAIエージェントによる自律的な組織**は多くの場合オーバーエンジニアリングです。
理由は次の通り:

- 作るのが一点物 = 各工程の判断が都度ユニークで、定型化しにくい
- 作り込みの良し悪しは美的判断が大半で、エージェント同士の自動受け渡しでは品質が安定しない
- 工程数が少なく、人がオーケストレーターとして十分回せる

そこでこのリポジトリでは、**「人がオーケストレーター、AIは工程ごとの専門アシスタント」** という
ゆるい役割分担から始めます。各役割は `agents/*.md` にプロンプトとして定義します。

## 役割一覧（アシスタントとして）

| ファイル | 役割 | 主なアウトプット |
|----------|------|------------------|
| `agents/concept-artist.md` | コンセプト/参照の提案 | バリエーション案、寸法・種別の提案 |
| `agents/modeler.md` | フォルム/シルエットのレビュー | ハイポリの改善点 |
| `agents/retopo-uv.md` | リトポ/UV 方針の助言 | ポリゴン配分・UVシーム案 |
| `agents/texture-artist.md` | マテリアル/PBR の助言 | マテリアル構成・PBR値 |
| `agents/qc-reviewer.md` | 規格・品質の最終チェック | 合否と修正指示 |

## いつ「組織化」に進むべきか

下の条件が **複数当てはまり始めたら**、エージェントを連携させた半自律パイプラインへの移行を検討します。

- ✅ 同種のアセットを **継続的に大量** に作るようになった
- ✅ 各工程の合否が **数値化** できている（ポリゴン数・スケール・命名など → 既に `standards.yaml` で一部達成）
- ✅ 人のレビュー時間がボトルネックになっている
- ✅ 工程間の受け渡しフォーマットが安定している

## 段階的な発展ロードマップ

```
Phase 0 (現在): 人がオーケストレーター + AIアシスタント役 + 自動検査
   ↓ 量が増え、基準が固まったら
Phase 1: QCレビューを半自動化（qc-reviewer が検査結果を読んで修正指示を生成）
   ↓ さらに定型化できたら
Phase 2: 工程間の自動受け渡し（concept→blocking のパラメータ自動生成など）
   ↓ 規模が production 級になったら
Phase 3: 複数エージェントのオーケストレーション（組織化）
```

各エージェントに割り当てるモデルの目安:

- 創造的提案（concept / modeler レビュー）: 最も高性能なモデル（Claude Opus 系）
- 規格チェック・整形（qc-reviewer の判定整形）: 速度重視の軽量モデル（Claude Haiku 系）でも可

> まずは Phase 0 を回し、ボトルネックが見えてから次の Phase に進むのが、
> 一点物制作では最もコスト効率が良いです。
